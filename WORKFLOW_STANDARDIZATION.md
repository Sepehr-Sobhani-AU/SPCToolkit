# Training Data Workflow - Standardized Approach

## Overview

The training data generation workflow has been standardized to follow a two-step modular approach:

1. **Step 1: Base Dataset Creation** - Create reusable, model-agnostic labeled data
2. **Step 2: Model-Specific Processing** - Generate training data with model-specific features

This design allows you to:
- Label data once, use for multiple models
- Experiment with different preprocessing parameters
- Maintain consistency between GUI and batch workflows

---

## Two Workflows, Same Output

### Workflow A: Interactive GUI-Based

**Use Case**: Interactive labeling and classification

1. **Load point cloud** → Use application to visualize
2. **Run clustering** → DBSCAN or other clustering plugins
3. **Classify clusters** → Manually classify or use ML classifier
4. **Export base dataset** → `Clusters > Export Classified Clusters` plugin
   - Output: `base_dataset/ClassName/classname_N.npy`
   - Format: (n, 6) arrays with [XYZ_centered, RGB]
   - Centered at origin (0,0,0)
   - Subsampled to max_points (default 20000)
5. **Generate training data** → `PointNet > Generate Training Data` plugin
   - Input: Base dataset from step 4
   - Output: Training-ready data with normals, eigenvalues, normalization

### Workflow B: Batch Processing

**Use Case**: Large datasets, pre-labeled PLY files

1. **Organize PLY files** by class:
   ```
   Dataset/
   ├── Tree/
   │   ├── tree_1.ply
   │   ├── tree_2.ply
   │   └── ...
   ├── Car/
   │   ├── car_1.ply
   │   └── ...
   └── ...
   ```

2. **Run batch conversion** → `python convert_ply_to_base_dataset.py`
   - Output: `base_dataset/ClassName/classname_N.npy`
   - Format: (n, 6) arrays with [XYZ_centered, RGB]
   - Centered at origin (0,0,0)
   - Subsampled to max_points (default 20000)

3. **Generate training data** → `PointNet > Generate Training Data` plugin
   - Input: Base dataset from step 2
   - Output: Training-ready data with normals, eigenvalues, normalization

## Base Dataset Format

### Specifications

**File Format**: NumPy binary (.npy)

**Array Shape**: `(n, 6)` where `n ≤ max_points` (default 20000)

**Feature Order**:
- Columns 0-2: X, Y, Z (centered at origin)
- Columns 3-5: R, G, B (normalized to [0, 1])

**Data Type**: `float32`

**Directory Structure**:
```
base_dataset/
├── metadata.json          # Dataset information
├── Tree/
│   ├── tree_1.npy        # (n₁, 6) array
│   ├── tree_2.npy        # (n₂, 6) array
│   └── ...
├── Car/
│   ├── car_1.npy
│   └── ...
└── ...
```

### Processing Details

1. **Centering**: Each cluster centered by subtracting its centroid
   - Reduces coordinate magnitude
   - Improves float32 precision
   - Makes features relative to cluster center

2. **Subsampling**: Clusters > max_points randomly subsampled
   - Default: 20,000 points
   - Ensures consistent processing time
   - Preserves spatial distribution

3. **Color Normalization**: RGB values in [0, 1] range
   - Standard Open3D format
   - Compatible with all models

## PointNet Training Data Format

### Specifications

Generated by `000_generate_training_data_plugin.py` from base dataset.

**File Format**: NumPy binary (.npy)

**Array Shape**: `(n×1024, num_features)`
- Default: `(1024, 9)` with normals and eigenvalues

**Feature Order** (with all options enabled):
- Columns 0-2: X_norm, Y_norm, Z_norm (normalized to unit sphere)
- Columns 3-5: Nx, Ny, Nz (normals)
- Columns 6-8: E1, E2, E3 (eigenvalues)

**Data Type**: `float32`

**Processing Pipeline**:
```
Base Dataset (XYZ+RGB, centered, subsampled to 20K)
    ↓
Load .npy file (n, 6)
    ↓
Extract XYZ only → discard RGB
    ↓
Create PointCloud object
    ↓
Normalize to unit sphere (center + scale)
    ↓
Compute normals on FULL cluster (before sampling to 1024)
    ↓
Compute eigenvalues on FULL cluster (before sampling to 1024)
    ↓
Stack features: [XYZ_norm, Normals, Eigenvalues]
    ↓
Sample to n×1024 points (default 1024)
    ↓
Save as training data
```

## Key Principles

### 1. Feature Computation on Full Data

**CRITICAL**: Normals and eigenvalues depend on neighborhood relationships.

✅ **CORRECT**:
```python
# Compute features on full cluster (up to 20K points)
point_cloud.estimate_normals(k=30)
eigenvalues = point_cloud.get_eigenvalues(k=30, smooth=True)

# THEN sample to 1024
indices = np.random.choice(n, 1024, replace=False)
features = combined[indices]
```

❌ **WRONG**:
```python
# Sample first
sampled_points = points[np.random.choice(n, 1024)]

# Then compute features (neighborhoods are incorrect!)
point_cloud = PointCloud(sampled_points)
point_cloud.estimate_normals(k=30)  # ← Wrong neighborhoods!
```

### 2. Preprocessing Consistency

Training and inference MUST use identical preprocessing:

| Step | Training | Inference |
|------|----------|-----------|
| Input | Base dataset .npy | Cluster from point cloud |
| Centering | Centered (in base dataset) | Center at origin |
| Subsampling | Random from ≤20K points | Random from ≤20K points |
| Features | Normals + eigenvalues on full | Normals + eigenvalues on full |
| Normalization | Unit sphere | Unit sphere |
| Sampling | Random to 1024 | Deterministic to 1024 (seed=42) |
| Output | (1024, 9) float32 | (1024, 9) float32 |

### 3. Float Precision

- **Load/Process**: Use float64 for precision during computation
- **Store**: Convert to float32 for storage efficiency
- **Consistency**: All saved arrays are float32

## Migration Guide

### From Old Workflow

If you previously used `convert_ply_to_training_data.py`:

**Old Approach** (single-step):
```
PLY files → convert_ply_to_training_data.py → Training data
```
- ❌ Computed features on subsampled data
- ❌ Not reusable for other models
- ❌ Preprocessing mismatch with inference

**New Approach** (two-step):
```
PLY files → convert_ply_to_base_dataset.py → Base dataset
           → Generate Training Data plugin → Training data
```
- ✅ Features computed on full data
- ✅ Reusable base dataset
- ✅ Consistent with inference

### Action Items

1. **Regenerate base dataset**:
   ```bash
   python convert_ply_to_base_dataset.py
   ```

2. **Generate training data**: Use plugin with base dataset

3. **Retrain model**: With new training data

4. **Validate**: Test on training data clusters (should work perfectly now)

## Configuration

### Base Dataset Conversion

Edit `convert_ply_to_base_dataset.py`:

```python
# Configuration (line ~360)
INPUT_DIR = r"C:\Users\Sepeh\OneDrive\AI\Point Cloud\_Data\Dataset"
OUTPUT_DIR = r"base_dataset"
MAX_POINTS = 20000  # Maximum points per cluster
MIN_SAMPLES_PER_CLASS = 3  # Minimum samples to keep a class
```

### Training Data Generation

Use plugin GUI parameters:
- **n_multiplier**: Points per sample (n × 1024), default: 1
- **normalize**: Enable unit sphere normalization, default: True
- **compute_normals**: Enable normal computation, default: True
- **normals_knn**: KNN for normals, default: 30
- **compute_eigenvalues**: Enable eigenvalue computation, default: True
- **eigenvalues_knn**: KNN for eigenvalues, default: 30
- **eigenvalues_smooth**: Smooth eigenvalues, default: True

## Troubleshooting

### Issue: "ValueError: setting an array element with a sequence"

**Cause**: Trying to use old training data with new code

**Solution**: Regenerate training data with new workflow

### Issue: Model fails on training data

**Cause**: Preprocessing mismatch

**Solution**:
1. Verify base dataset format: `(n, 6)` with float32
2. Check training data format: `(1024, 9)` with float32
3. Run diagnostic: `python diagnose_mismatch.py path/to/cluster.ply`

### Issue: Most clusters skipped

**Cause**: Clusters have < 1024 points after base dataset creation

**Solution**:
- Reduce `n_multiplier` to 0.5 (512 points) in training data generation
- Or ensure source PLY files have enough points

## Benefits of This Approach

### Modularity
- Separate base data from model-specific preprocessing
- Easy to add new models (RandomForest, DGCNN, etc.)
- Each model plugin processes the same base dataset

### Efficiency
- Label data once, reuse forever
- Experiment with different feature combinations
- No need to re-export when changing preprocessing

### Consistency
- Both GUI and batch workflows produce identical output
- Training and inference use matching preprocessing
- Reduced risk of bugs from pipeline differences

### Maintainability
- Clear separation of concerns
- Easy to debug (inspect intermediate base dataset)
- Version control friendly (metadata tracks parameters)

## Future Extensions

The standardized base dataset format enables:

1. **Additional Models**:
   - RandomForest plugin (use different features)
   - DGCNN plugin (graph-based features)
   - PointNet++ plugin (hierarchical features)

2. **Feature Experimentation**:
   - Height above ground
   - Intensity values
   - Geometric features (planarity, sphericity)
   - Color histogram features

3. **Augmentation**:
   - Random rotations
   - Scaling variations
   - Point dropout
   - Noise injection

All extensions work with the same base dataset format!
